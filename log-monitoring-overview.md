## 2. ログの 2 系統 — ここが最も重要

現行システムのログは**用途が全く異なる 2 系統**に分かれる。これを混同すると設計が破綻する。

```
┌──────────────────────────────────────────────────────────┐
│                    ログの 2 系統                          │
├──────────────────────────┬───────────────────────────────┤
│  系統A: 実績連動ログ       │  系統B: 運用・監視ログ          │
│  （業務データファイル）      │  （観測・障害調査用）            │
├──────────────────────────┼───────────────────────────────┤
│ AP → 加工 → 分割 → 転送   │ 各サーバー → 収集 → 集約 → 保管 │
│ → Nx ホスト → 後続バッチ   │ → 世代管理 → 検索・分析         │
├──────────────────────────┼───────────────────────────────┤
│ バッチ処理として実行        │ 常駐エージェントが収集           │
│ JP1 ジョブネットで制御      │ JP1 + cron で定期取得           │
├──────────────────────────┼───────────────────────────────┤
│ 欠損 = 業務停止            │ 欠損 = 調査困難（業務は動く）    │
└──────────────────────────┴───────────────────────────────┘
```

→ 詳細: [ログ基本設計書](log-design.md)

## 4. 何が常駐で、何がバッチか

![常駐収集とバッチ処理](diagrams/log-collection-vs-batch.png)

- **集める** のは常駐処理（Prometheus / Promtail）
- **加工して外へ渡す** のはバッチ処理（Tekton Pipeline）
- 監視基盤はバッチを**観測する側**、バッチは**観測される側**

→ 詳細: [ログ基本設計書 セクション 5](log-design.md#5-収集は常駐加工転送はバッチ)
